# Worm

iOS app (SwiftUI, iOS 17+). A **personality / taste profile** represented by a
worm mascot, assembled from maximal-extraction data **nodes** (Spotify, Apple
Music, YouTube, Contacts, Photos, …).

## Read this first

**[`docs/vision.md`](docs/vision.md) is the source of truth** for what this app
is, the tone, the rules, and the roadmap. Read it before building anything
product-facing. The short version:

- The **worm is the taste profile** — it knows the user uncannily well, and
  proves it in one specific, surprising, *true* sentence.
- The first product use case is **taste -> music**, not music -> music. Music
  recommendation must use the whole taste profile: YouTube, contacts, photos,
  calendar, media, routines, aesthetics, places, and future nodes, not just
  listening history.
- **Anti-slop is the bar.** Insights are *observed*, never summarized. No
  horoscopes, no "you have eclectic taste," no feature-lists of someone's own
  data. If it could be said about anyone, cut it.
- **Nodes squeeze every drop** the source exposes, and **persist it** — connect
  once, set up forever; refresh is a light top-up, never a full re-sync.
- **First delight is fast and partial** — the first "wow" lands seconds after
  the first node connects, off whatever has synced so far.
- Three layers: **nodes** (raw data) → **brain slices / taste profile**
  (synthesis + query boundary) → **worm** (voice).

## Build & run

The Xcode project is generated by [XcodeGen](https://github.com/yonkeltron/XcodeGen)
from `project.yml`. `Worm.xcodeproj` is regenerable and **not** the source of
truth — after adding/removing/renaming files, regenerate:

```sh
xcodegen generate
xcodebuild -project Worm.xcodeproj -scheme Worm \
  -destination 'generic/platform=iOS Simulator' \
  -configuration Debug build CODE_SIGNING_ALLOWED=NO
```

Editor (SourceKit) diagnostics often show stale "cannot find X in scope" /
"No such module UIKit" errors right after adding files or because indexing runs
against macOS — trust the `xcodebuild` result over them.

## Layout

- `App/` — `WormApp` (owns the nodes), `RootView` (nav stack), `HomeView`
  (greeting-only home), `DevFlags`.
- `Mascot/` — splash, inchworm, the home greeting (`WormGreeting` is node-free
  and permission-free), Liquid Glass helper.
- `Graph/` — the dev node-graph (behind `DevFlags.showGraphButton`).
- `Nodes/` — one folder per node; each is an `@Observable` manager + detail
  view, persisting a full snapshot via `Persistence/SnapshotStore`.
- `Profile/` — brain slices, deterministic retrieval, `TasteProfile`,
  `BrainSynthesizer`, debug chat, and the profile surface. Nodes feed slices;
  only the brain talks to Claude.

Adding a node: create its `@Observable` manager + detail view, add a `NodeRoute`
case (wired in `RootView`), add it to `PersonalityGraphView.nodeList`, inject it
in `WormApp`, and add any usage-description key to `Info.plist`.

## Conventions

- Match the surrounding code's idiom, naming, and comment density. New nodes
  mirror the existing node lifecycle exactly (`connect` / `disconnect` /
  `syncEverything` / `restoreSessionIfPossible`, cached snapshot, no auto
  re-sync once a snapshot exists).
- Capture everything a source offers — don't trim the data surface to what a
  current screen happens to show.
- Keep Claude behind the brain layer (`BrainRetriever` -> `BrainSynthesizer` ->
  `ClaudeClient`). Node managers and feature extractors should produce compact
  slices or facts, never call model clients directly.
- For music discovery, Claude never gets to invent catalog items. Build or
  retrieve real candidates, verify them against a catalog, then use the brain to
  explain and rank. No verified track, no surfaced recommendation.
